AGENTIC AI SECURITY

Securing autonomous agents — an AI that doesn't just answer but plans, calls tools, and acts (so every action is new attack surface)
CTF Farooq Ali
· security explainer series
v2026.06
01
Why agents change the game — it doesn't just answer, it acts

An agent plans, calls tools, keeps memory, and takes actions on its own — every one of those is new attack surface a chatbot never had.

02
The agent attack surface — the loop, and where it's attacked

Every node is a target — and the output of one feeds the next, so one compromise steers the whole loop.

03
Tool abuse

An agent tricked — usually by prompt injection — into calling a dangerous tool it has access to:

📧 send email💻 run code💸 move money🗑️ delete data

The tool did exactly what it was told — by the wrong instruction.

04
Excessive agency — OWASP LLM06
🧩
Too much functionality — tools it never needed.
🔓
Too many permissions — broad scopes, standing access.
🕶️
Too little oversight — acts with no approval.

The root cause of most agent harm — the damage a hijacked agent can do.

05
Memory poisoning

Hostile content is written into persistent memory — then resurfaces later to steer the agent:

plant instruction→ stored as memory→ recalled later

A one-time attack that keeps acting — long after the attacker has gone.

06
Cross-agent trust failure

In a multi-agent system, one agent blindly trusts another's output as if it were fact or command:

Agent A (compromised)→ Agent B acts on it

Trust between agents must be verified — an agent is not a trusted user.

07
Prompt injection → action — the confused deputy, now with your privileges

For a chatbot, injection corrupts text. For an agent, the model's output drives real tool calls — so injection becomes a remote action run with the user's privileges.

08
Controls — treat the agent like a privileged identity

🔑 Least privilege

Scope every tool. No standing access it doesn't need right now.

✋ Approval gates

Human-in-the-loop for sensitive / irreversible actions.

🪪 Privileged identity

Its own scoped credentials — authenticated, authorized, audited.

↩️ Reversibility

Prefer undoable actions. Keep a kill switch to stop the loop.

👁️ Observability

Log every decision, tool call, and action — trace & alert.

🎯 Threat-model it

Model planner · memory · tools · output as distinct surfaces.

Constrain what the agent CAN do — don't rely on it choosing not to.

09
On the wire — gate the call
// tool the agent may invoke — narrowly scoped
{ "tool": "refund", "max": 50, "scope": "orders:own" }
// over the limit → not auto-run, it's gated
{ "action": "refund $9000", "approval": "human" }

The platform enforces the limit — never the model's good intentions.

10
Watch out
🚨
Autonomy on irreversible acts — no approval gate = one bad step, no undo.
🔓
Over-broad tool scopes — every extra permission widens the blast radius.
🔁
Unbounded loops / cost — cap steps, spend, and time. Watch for runaway plans.
🎭
Trusting agent output as truth — it's suggestible input, not a verified fact.
11
The takeaway
Treat an agent like a powerful new employee with system access — give it least privilege, human approval on real actions, a full audit trail, and a kill switch.
Least privilege→ Approve real actions→ Audit everything→ Kill switch
