OWASP API TOP 10

The 2023 edition — the ten risks that break APIs, where authorization failures dominate (APIs expose objects & functions directly, so ids and permissions are the attack surface)

01
Why APIs are their own risk surface — no UI to hide behind

An API hands the caller object- and function-level operations directly — so the classic web bugs give way to broken authorization as the number-one problem.

02

#RiskIn one line

Notice API1, API3 & API5 — three of the top five are authorization failures.

03
Spotlight · API1 BOLA — the #1 API problem: change the id, get someone else's data

The token proves who you are — it does not prove the object is yours. Authorize the object, per request.

04
Authorization is the theme — three of the ten, one root cause

API1 · BOLA (object)

Can I read this object? Change the id, reach another user's record./orders/1043

API3 · BOPLA (property)

Can I read/write this field? Mass-assign role; leak hidden fields.{"role":"admin"}

API5 · BFLA (function)

Can I call this action? Hit an admin route as a normal user.DELETE /admin/users

Object · property · function — authorize all three, on every request, server-side.

05
Web bug ≠ API bug
🖥️
Classic web — XSS, CSRF, injection through a rendered UI the server controls.
🔑
APIs — the client sends ids & fields directly, so authorization is the top risk.

Same OWASP mindset, different weak spot — the OWASP API Top 10 is its own list.

06
API9 · what you forgot
👻
Shadow API — live but undocumented; never security-reviewed.
🧟
Zombie API — an old /v1 still running, unpatched, forgotten.

You can't defend what you don't know you're running — inventory every host, version & endpoint.

07
How to defend — the moves that shut down most of the list
Authorize every object & function→ Allow-list input fields→ Rate & size limits
Inventory every API + version→ Validate outbound URLs (SSRF)→ Distrust third-party data

Object-level checks alone would neutralize the #1 risk (API1 BOLA).

08
On the wire — one changed id is the whole exploit
// authenticated as Alice (user 1042) — the token is valid…
GET /api/orders/1042   Authorization: Bearer eyJhbGciOi…  // her own order ✓
// …but nothing stops her asking for the next id — BOLA
GET /api/orders/1043   Authorization: Bearer eyJhbGciOi…  // Bob's order 🔥 200 OK
// the fix: on every access, the server checks ownership
if (order.owner_id != token.sub) return 403;
09
API4 & API7
💸
Unrestricted consumption — no rate/size caps means a big bill or a downed service.
🌐
SSRF — pass a URL, the server fetches it — into cloud metadata & internal hosts.

Cap requests, payloads & cost; validate and allow-list any URL your server will fetch.

10
Watch out
🎭
Auth ≠ authz — a valid token doesn't mean this object or action is theirs.
🕵️
Client-side gates — hiding a button isn't security. Enforce on the server.
📮
Mass assignment — bind requests to a field allow-list, never the whole object.
🗺️
Untracked endpoints — old versions & test hosts are the softest way in.
11
The takeaway
The OWASP API Top 10:2023 has one loud theme — broken authorization. APIs expose objects & functions directly, so check ownership and permission on every request.
Who are you?→ Is this object yours?→ May you call this?→ Only then serve
